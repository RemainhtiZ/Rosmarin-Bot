# 自动生产执行链路（资源调度 → 触发生产 → 收回产物）

---

## 设计准则（本文档与实现以此为准）

1. 资源管理先在全局判断“哪些资源可以支持生产某个产物”，再把产物写入各房间的生产计划；计划分配保持均衡（避免有的房间计划一大堆，有的房间只有几个），并安排对应资源调度。
2. 房间自动生产先从生产计划中选“当前就能生产的一项”（房间内资源足够至少生产 1 次）开始生产；若没有可生产项，再走兜底默认生产配置；都没有则闲置。
3. 跨房资源调度避免太远：若传输能量成本 `cost > amount`，则不进行该组调度。

## 0. 两条产线 + 一条调度线

```mermaid
flowchart LR
  RM[ResourceManage\n(跨房调度 + 可选排产)] -->|1 写计划| AutoData[AutoLabData / AutoFactoryData]
  RM -->|2 注入原料需求阈值| Thresholds[ThresholdMap/ResManageMap]
  RM -->|3 下发发送任务| Send[MissionPools: terminal/send]

  Send --> TW[TerminalWork\n执行 terminal.send]
  TW --> RM

  AutoData --> AL[autoLab / autoFactory\n选择当前任务(StructData)]
  AL --> Move[Transport / Manage\n搬运补料/清料]
  Move --> Work[LabWork / FactoryWork\n实际生产]
  Work --> Move2[Manage(回收产物)]
  Move2 --> Stock[storage/terminal\n库存]
```

## 1. 关键数据：计划 / 当前任务 / 调度任务

### 1.1 计划（Plan）：资源管理写入的“要生产到多少”

- `Memory.RosmarinBot.AutoData.AutoLabData[room][product] = limit`
- `Memory.RosmarinBot.AutoData.AutoFactoryData[room][product] = limit`

含义：该房间应把 `product` 生产到 `limit`（上限）。这不是立即执行命令，而是给 autoLab/autoFactory 的“候选任务列表”。

### 1.2 当前任务（Active Task）：autoLab/autoFactory 写入的“本房间现在做什么”

- Lab：`Memory.RosmarinBot.StructData[room].labAtype / labBtype / labAmount`
- Factory：`Memory.RosmarinBot.StructData[room].factoryProduct / factoryAmount / factoryLevel`

含义：这是 **执行侧**（LabWork/FactoryWork、搬运任务生成器）真正依赖的状态。

### 1.3 调度任务（Logistics）：跨房发送 + 房内搬运

跨房发送（由资源管理下发，终端执行）：

- `Memory.RosmarinBot.MissionPools[sourceRoom].terminal[]` 中 `type='send'`
- `data = { targetRoom, resourceType, amount }`

房内搬运（由 manage/transport 任务生成器下发，creep 执行）：

- `MissionPools[room].manage[]`：搬运到/搬运出 factory、补终端等
- `MissionPools[room].transport[]`：补 lab、清 lab、boost 相关

## 2. 资源管理与资源调度（排产 → 调度 → 收敛）

### 2.A 排产：资源管理如何写入生产计划（AutoData）

资源管理的排产职责是：把“要生产什么、生产到多少”写进 `AutoLabData/AutoFactoryData`，并把所需输入资源写进需求阈值，从而驱动跨房调度把原料送到生产房间。

#### 2.A.1 计划写入点（AutoData）

- Lab：`AutoLabData[room][product] = limit`
- Factory：`AutoFactoryData[room][product] = limit`

其中 `limit` 是“目标库存上限”（通常用“当前库存 + batch”形式下发，避免一次写得过大）。

#### 2.A.2 全局可生产性过滤（只下发可达任务）

为避免“下发了计划但全局根本没有任何输入资源”，资源管理会先做过滤：

- Lab：只有当产物缺口存在，且配方底物 `raw1/raw2` 在全局可消耗（不会把底物压到需求线以下）时，才下发计划。
- Factory：候选商品除了等级/白色链/缺失 deposit 根资源过滤外，还要求配方 `components` 的每个 component 全局可用量 > 0；若存在任何 component 全局为 0，则该商品不会下发计划。

#### 2.A.3 计划分配的均衡策略（避免计划集中在少数房间）

资源管理不会对一个房间塞很多条计划，而是按“每房间最多 1 条链条计划”的方式分配，并采用“候选列表 + 房间轮转”的策略做到均衡：

- 先构造候选产物列表（按优先级/缺口排序）
- 再遍历可下发的房间列表，每个房间从候选列表轮转取 1 项下发
- 房间若已有正在生产任务或已有未完成计划，则跳过（避免覆盖与刷屏）

这样可以避免“同一轮所有房间扎堆生产同一个产物”，也避免“有的房间计划堆积、有的房间没计划”。

#### 2.A.4 输入资源需求注入（驱动调度）

资源管理会把生产所需输入资源注入需求阈值：

- Lab：把 `raw1/raw2` 的最低拉取量写入该房间需求阈值
- Factory：把 `components` 的最低拉取量写入该房间需求阈值

后续跨房调度会把这些输入资源当作“必须补齐”的目标。

### 2.B 调度：ResourceManage → terminal/send → TerminalWork

每轮资源管理会为 eligible 房间建立每种资源的阈值：

- 默认来自 `RESOURCE_BALANCE[res] = [targetThreshold, sourceThreshold]`
- 可用 `Memory.RosmarinBot.ResourceManage[room][res]` 覆盖

判断规则：

- `amount < targetThreshold` → 该房间需要补（target）
- `amount > sourceThreshold` → 该房间可供给（source）

### 2.2 去重：避免重复下发同一类 send

ResourceManage 会扫描全局所有 `terminal/send` 任务，统计每房间：

- `queuedOut[room][res]`：已排队待发出
- `queuedIn[room][res]`：已排队待收到

缺口/盈余计算会把 queuedIn/queuedOut 算进去，所以 **不会每 50 tick 叠加同一类发送任务**。

### 2.3 输出：只写任务，不直接 send

匹配到 source→target 后，ResourceManage 只做一件事：

- 对 sourceRoom 写入/扩容 `terminal/send` 任务（UpsertMax：只增不减）

执行交给 `TerminalWork`：

- 读取 send 任务
- 按手续费与最小批量分批 `terminal.send`
- 成功则递减 amount；为 0 删除任务

### 2.4 成本规则：cost > amount 则跳过

跨房调度会估算并计算 `terminal.send` 的能量手续费（transaction cost）。当满足：

- `cost > amount`

则认为该调度“距离过远/成本过高”，本轮不下发该 source→target 的发送任务。

### 2.5 这个闭环怎么“收敛”

send 任务执行后，资源实际进入 targetRoom 的 terminal/storage（后续可能再被搬走）。

下一轮 ResourceManage 用新库存 + queuedIn/queuedOut 重算缺口：缺口变小 → send 下发变少/停止。

## 3. 自动 Lab：缺什么 → 触发合成 → 产物入库

### 3.1 触发生产（从调度到“开始合成”）

有两种方式让房间进入“可合成”状态：

1) 手动：你在 StructData 里直接设定 `labAtype/labBtype`（或通过控制台命令）
2) 自动（推荐）：ResourceManage 写入 AutoLabData（生产计划），autoLab 选中后写 StructData

自动链路关键点：

- ResourceManage 选定要生产的 lab 产物 `product` 后：
  - 写 `AutoLabData[room][product] = 当前库存 + batch`
  - 把该产物配方的 `raw1/raw2` 作为“最低需求阈值”注入资源调度体系  
    这一步会触发跨房 send，把原料搬到生产房间（terminal → storage）。

### 3.1B autoLab 的任务选择顺序（计划优先）

autoLab 的决策顺序固定为：

1. 从 `AutoLabData[room]`（生产计划）中筛选“当前可生产”的任务：  
   - 未达上限（`current < limit`，带一定容忍，比如 0.9）
   - 底物在房间可用量充足（至少能生产 1 次，且满足 inputMin）
2. 若计划里没有可生产项，再走兜底默认合成（T1 → T2 → T3）
3. 仍找不到任务则闲置

边界处理（避免误停工/崩溃）：

- `labAmount` 可能为空：按 0 处理（表示“无上限”或“未配置上限”），避免误判后清空任务。
- `AutoLabData` 中若出现不在 `LabMap/LabLevel` 的键：会被跳过，不会导致任务选择异常。

### 3.2 autoLab 负责“把计划变成当前任务”

autoLab 读取 `AutoLabData[room]`，结合可用量判断是否需要继续生产：

- 可用量口径包含 `lab.store`（避免“原料已搬进 lab 但统计不到”导致误判）
- 不达限额但缺料时，会进入等待窗口（避免频繁清任务）

最终写入：

- `StructData.labAtype/labBtype/labAmount`

### 3.3 补料与执行：TransportMission + LabWork

补料/清料：

- `transportMission` 会把 raw1/raw2 从 storage/terminal 搬进 A/B lab
- 若 A/B lab 装错矿，会先清回 storage

执行：

- `LabWork/RunReaction` 周期性对非 A/B 的反应 lab 执行 `runReaction`

### 3.4 产物如何“收回入库”

产物默认会产在反应 lab 里，后续由搬运体系（transport/manager）把产物转运回 storage/terminal。

当库存增长到 AutoLabData 的 `limit` 附近时：

- autoLab 认为目标已达成 → 结束/切换任务
- ResourceManage 下一轮全局缺口变小 → 不再下发该产物计划

## 4. 自动 Factory：资源调度 → 自动生产 → 回收产物

### 4.1 触发生产：四色链条排产（可选）+ components 注入调度

当 `RESOURCE_PRODUCTION.factory.chain.enabled`：

- ResourceManage 会按房间有效等级挑选要生产的 `product`：
  - 等级匹配：`room.factory.level || StructData.factoryLevel`
  - 白色商品不自动合成（排除 composite/crystal/liquid 链）
  - 缺失 deposit 根资源（metal/biomass/silicon/mist）时跳过对应颜色链
  - `keepByLevel` 防止无限堆积：达到目标库存后切换同级其他颜色或停止

选中 product 后：

- 写 `AutoFactoryData[room][product] = 当前库存 + batch`
- 把 `COMMODITIES[product].components` 作为“最低需求阈值”注入资源调度体系  
  这一步会触发跨房 send，把组件调度到生产房间的 terminal/storage。

### 4.1B autoFactory 的任务选择顺序（计划优先）

autoFactory 的决策顺序固定为：

1. 从 `AutoFactoryData[room]`（生产计划）中筛选“当前可生产”的任务：  
   - 未达上限（`current < limit`，带一定容忍，比如 0.9）
   - 配方 `components` 在房间可用量足够至少生产 1 次
   - 同时考虑缺口优先（`limit-current` 越大越优先）
2. 若计划里没有可生产项，再走兜底默认生产：基础商品 → zip →（可选）链条兜底
3. 仍找不到任务则闲置

边界处理（避免兜底异常）：

- `AutoFactoryData` 中若出现不在 `COMMODITIES` 的键：会被跳过，不会崩溃。
- zip 兜底：若房间没有 mineral 或该 mineral 没有对应压缩目标，则直接不触发。
### 4.2 autoFactory：把计划变成当前任务


autoFactory 读取 `AutoFactoryData[room]`（以及一些兜底任务），用“可用量口径包含 factory.store”做判断：

- 够生产至少 1 次且未达限额 → 保持当前任务
- 缺料则等待一段时间（避免频繁换任务）

最终写入：

- `StructData.factoryProduct/factoryAmount`

### 4.3 补料、清杂、回收：ManageMission

ManageMission 会围绕 factory 维护“可生产状态”：

- 把 components 从 storage/terminal 搬进 factory
- 把非材料/非产物的杂物从 factory 搬走
- 产物在 factory 里堆积到阈值后搬出到 storage/terminal

### 4.4 生产执行：FactoryWork

当以下条件满足时调用 `factory.produce(product)`：

- factory 存在且 cooldown==0
- `StructData.factory` 开启，且 `factoryProduct` 存在
- `factory.store` 中 components 足够

### 4.5 产物收回与闭环收敛

FactoryWork/ManageMission 会把产物从 factory 搬回 storage/terminal。

库存上升后：

- autoFactory 达到限额会结束/切换任务
- ResourceManage 下一轮看到全局缺口减少，不再下发该产物计划
